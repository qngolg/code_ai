# AI Agent 任务执行模板 (Task Template)

> **最后更新**: 2026-04-26
> **维护说明**: 当 Agent 因任务分解不合理而引入 Bug 或架构偏差时，必须更新本文档，增加新的约束或场景。

## 核心指令：你必须像一名严谨的工匠那样工作

本项目遵循 **TDD + 小步提交 + 持续验证** 的工作流。当你收到一个开发任务时，**绝对禁止**一次性生成所有代码然后宣布完成。你必须严格遵循以下 6 个阶段，任何阶段的失败都必须回到该阶段重新开始，不得跳步。

---

## 任务执行 6 阶段模型

### 阶段 1：任务分析与规划
**你的输出**：一份简短的作战计划。

- 仔细阅读任务描述，识别涉及的 **DDD 分层**（接口、应用、领域、基础设施）。
- 列出需要创建或修改的**领域实体/值对象**、**领域服务**、**应用服务**、**仓库接口**等。
- 将大任务切分为多个**独立的子任务**，每个子任务聚焦单一概念。
- 如果任务超过 3 个文件或 50 行逻辑，必须拆分为子任务逐一交付。
- 计划中必须明确**第一个子任务的失败测试**将如何写。

**示例输出**：
```markdown
子任务 1：实现“订单创建”核心领域逻辑
+ 新建 Order 聚合根（Domain）
+ 新建 OrderRepository 接口（Domain）
+ 编写 OrderSpec 测试（Domain）
+ 关键测试：库存不足时抛出 InsufficientStockException

子任务 2：实现订单创建的应用编排
+ 新建 CreateOrderCommand (Application)
+ 新建 OrderApplicationService (Application)
+ 编写 OrderApplicationServiceSpec 测试
```

### 阶段 2：编写失败的测试（红灯）
**你的输出**：一个仅包含测试文件且编译失败的 commit。

- 严格按照 `docs/testing-guide.spock.md` 的模板编写 **Spock 测试**。
- 测试必须覆盖：**正常路径**、**失败路径**、**边界条件**。
- 此阶段**不能**创建或修改任何实现类。
- 提交信息格式：`FAILING TEST: [子任务描述]`。
- 推送该 commit，**等待 CI 验证测试确实失败**。如果 CI 报告测试意外通过，则此次任务 **立即中止**，你需要反思测试的有效性并重写。

### 阶段 3：最小实现（绿灯）
**你的输出**：刚好让测试通过的最少代码，一个干净的 commit。

- 根据阶段 2 的失败测试，仅编写让测试通过的实现代码。
- **严格**遵守 `docs/architecture-guide.md` 中的分层依赖规则。
- 禁止进行“顺手重构”或添加无关功能。
- 代码写完后，运行 `./scripts/review.sh` 进行本地 AI 自我审查，修正其反馈的问题。
- 本地 `gradle test` 全部通过后，提交代码。提交信息格式：`IMPLEMENT: [子任务描述]`。

### 阶段 4：重构与审查（清洁）
**你的输出**：一个仅涉及结构优化的独立 commit。

- 在测试保护下，审视阶段 3 的代码，进行必要重构：
    - 消除重复。
    - 改善命名，使其体现业务意图。
    - 确保代码位于正确的包内。
- **绝不允许**在重构时修改业务行为。
- 重构后再次运行全部测试，通过后提交。提交信息格式：`REFACTOR: [重构说明]`。
- 如有任何架构层面的疑问，必须查阅 `AGENTS.md` 指向的文档，而非猜测。

### 阶段 5：端到端验证
**你的输出**：确认功能在所有层次正确集成的证据。

- 如果子任务涉及跨层协作（例如从 Controller 到数据库），必须编写或执行一个基本的 **集成测试** (`*IT.groovy`) 或运行已有的集成测试套件。
- 对于 Web 接口，可使用 `curl` 命令验证真实响应（你可以在终端中执行命令）。
- 必须确保 **MyBatis-Plus 操作实际生效**，不仅仅是 mock 通过。
- 所有集成测试通过后，子任务才算真正完成。

### 阶段 6：文档同步与宣布完成
**你的输出**：更新相关文档，并明确声明任务结束。

- 检查本次改动是否需要更新：
    - `docs/architecture-guide.md`（新增了重要领域概念？）
    - `docs/testing-guide.spock.md`（发现了新的测试模式？）
    - `AGENTS.md`（本任务促进了新的使用模式？）
- 在 PR 描述或任务评论中，列出所有完成的子任务、通过的测试数量以及任何已知限制。
- 只有完成本阶段后，才能报告“任务完成”。

## 绝对约束与红线

| 红线 | 后果 |
| :--- | :--- |
| 在阶段 2 之前编写任何实现代码 | PR 被拒绝，任务重置 |
| 提交的代码导致 ArchUnit 约束测试失败 | CI 阻断，你必须根据失败日志修正分层违规 |
| 测试中未包含负面场景（异常、空值等） | 反馈循环强制重写测试 |
| 一个 commit 同时包含测试和实现 | 任务被打回，必须拆分为至少两个 commit |
| 自创不在 `AGENTS.md` 或相关文档中的架构模式 | 视为架构漂移，任务暂停 |

## 示例任务：添加“用户积分扣减”功能

**任务入口**：人类项目经理在 Issue 中描述：“实现积分扣减，用于下单时抵扣金额。”

**Agent 执行流程示例**：

1.  **阶段 1**：输出计划 → 拆分为：
    - 子任务 1.1：领域服务 `PointsDeductionDomainService` 逻辑
    - 子任务 1.2：应用服务编排
2.  **阶段 2 (子任务 1.1)**：创建 `PointsDeductionDomainServiceSpec.groovy`，编写测试：
    - “积分足够时，扣减成功返回剩余积分”
    - “积分不足时，抛出 `InsufficientPointsException`”
    - 提交 `FAILING TEST: Add points deduction domain logic`。
3.  **阶段 3**：实现 `PointsDeductionDomainService.java`，通过测试。提交 `IMPLEMENT: Points deduction domain logic`。
4.  **阶段 4**：重构方法名，使其更清晰。提交 `REFACTOR: Rename method to deductPoints`。
5.  **阶段 5**：运行集成测试 `OrderApplicationServiceIT`，确认下单流程中积分扣减真实生效。
6.  **阶段 6**：未发现需更新的文档，报告子任务 1.1 完成。继续子任务 1.2。
